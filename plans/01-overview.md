## 개요

### 프로젝트 정보
- **프로젝트명**: 위로하는 일기 앱 (Comfort Journal App)
- **목적**: GitHub 페이지를 통해 프로젝트를 소개하고, 주요 기능과 가치를 전달
- **타겟 플랫폼**: GitHub Pages (정적 사이트)
- **예상 개발 기간**: 1주일

### 프로젝트 핵심 가치
> "힘든 일상을 견디는 사람들이 일기조차 부담스러운 순간에, 감사를 찾으라고 요구하는 기존 앱들과 달리, 당신이 여기 온 것만으로도 충분하다고 말하며 진정한 위로를 제공한다"

## 목적 및 목표

### 주요 목적
1. **프로젝트 소개**: 프로젝트의 목적과 핵심 가치 전달
2. **기능 시연**: 주요 기능을 시각적으로 보여주기
3. **기술 스택 소개**: 사용된 기술과 아키텍처 공유
4. **컨트리뷰션 유도**: 오픈소스 기여자 유도 (선택적)

### 구체적 목표
- [ ] 프로젝트의 차별점을 명확히 전달
- [ ] 사용자 여정(User Journey)을 시각적으로 표현
- [ ] 주요 기능 3-5개를 인터랙티브하게 소개
- [ ] 기술 스택과 아키텍처를 간단히 설명
- [ ] 반응형 디자인으로 모바일/데스크톱 모두 지원
- [ ] 접근성(Accessibility) 고려
- [ ] 빠른 로딩 속도 (Lighthouse 점수 90+)

## 타겟 오디언스

### 주요 타겟
1. **개발자 커뮤니티**
   - 오픈소스 기여 관심 있는 개발자
   - 비슷한 프로젝트를 만들고 있는 개발자
   - 기술 스택을 학습하려는 개발자

2. **잠재적 사용자**
   - 일기 앱에 관심 있는 사람
   - 정신 건강/웰빙 앱을 찾는 사람
   - AI 기반 앱에 관심 있는 사람

3. **기업/조직**
   - 비슷한 프로젝트를 기획 중인 기업
   - 기술 파트너십을 찾는 조직

### 오디언스별 기대 행동
- **개발자**: GitHub 저장소 방문, 코드 리뷰, 기여
- **일반 사용자**: 프로젝트 이해, 데모 체험, 피드백
- **기업**: 프로젝트 평가, 기술 문의, 협업 제안

## 페이지 구조

### 1. Hero 섹션
- **목적**: 첫인상, 핵심 메시지 전달
- **콘텐츠**:
  - 프로젝트명/로고
  - 한 줄 소개: "감사를 강요하지 않는 유일한 일기 앱"
  - CTA 버튼: "데모 보기" / "GitHub에서 보기"
- **디자인**: 큰 타이포그래피, 부드러운 배경 그라데이션

### 2. 문제 정의 섹션 (Problem Statement)
- **목적**: 프로젝트가 해결하려는 문제 설명
- **콘텐츠**:
  - 기존 일기 앱의 한계점
  - 사용자 인터뷰에서 발견한 주요 불편함
  - 차별점 강조
- **디자인**: 인포그래픽, 통계 데이터 시각화

### 3. 핵심 기능 섹션 (Features)
- **목적**: 주요 기능 3-5개 소개
- **기능 목록**:
  1. **공감하는 AI 동반자**: 감정을 이해하고 위로하는 AI
  2. **부담 없는 입력**: 30초 안에 끝나는 간편 입력
  3. **생애주기별 맞춤**: 사용자 상황에 맞는 질문과 응답
  4. **칭찬 시스템**: 시도한 것 자체를 칭찬
  5. **유연한 감사 찾기**: 강요하지 않는 감사 발견
- **디자인**: 카드 레이아웃, 아이콘, 스크린샷/일러스트

### 4. 사용자 여정 섹션 (User Journey)
- **목적**: 사용자가 앱을 사용하는 과정을 시각화
- **콘텐츠**:
  - 스텝별 플로우 (온보딩 → 일기 쓰기 → 위로 받기)
  - 실제 대화 예시
  - 감정 변화 시각화
- **디자인**: 타임라인, 인터랙티브 플로우차트

### 5. 기술 스택 섹션 (Tech Stack)
- **목적**: 사용된 기술과 아키텍처 소개
- **콘텐츠**:
  - Frontend: React, TypeScript, Tailwind CSS, shadcn/ui
  - Backend: Express, Node.js
  - AI: Anthropic Claude API
  - Database: PostgreSQL (Neon)
  - Deployment: Vercel (또는 GitHub Pages)
- **디자인**: 로고 그리드, 아키텍처 다이어그램

### 6. 데모/스크린샷 섹션
- **목적**: 실제 앱 화면 보여주기
- **콘텐츠**:
  - 주요 화면 스크린샷 (홈, 대화, 타임라인)
  - 인터랙티브 데모 (선택적)
  - GIF/비디오로 동작 보여주기
- **디자인**: 캐러셀, 라이트박스 갤러리

### 7. 통계/성과 섹션 (선택적)
- **목적**: 프로젝트 성과 공유 (있는 경우)
- **콘텐츠**:
  - 사용자 수 (있다면)
  - 만족도 점수
  - 주요 메트릭
- **디자인**: 숫자 카운터 애니메이션, 차트

### 8. CTA 섹션 (Call to Action)
- **목적**: 다음 행동 유도
- **콘텐츠**:
  - "GitHub에서 보기" 버튼
  - "데모 체험하기" 버튼 (있는 경우)
  - "기여하기" 버튼 (오픈소스인 경우)
- **디자인**: 큰 버튼, 명확한 액션

### 9. 푸터 (Footer)
- **목적**: 추가 정보 및 링크
- **콘텐츠**:
  - 프로젝트 정보
  - 라이선스
  - 연락처/이슈 트래커 링크
  - 소셜 미디어 링크 (있는 경우)

## 성공 지표

### 정량적 지표
- [ ] 페이지 로딩 속도: Lighthouse 점수 90+ (Performance, Accessibility, Best Practices, SEO)
- [ ] 반응형 디자인: 모바일/태블릿/데스크톱 모든 화면 크기 지원
- [ ] 접근성: WCAG AA 기준 충족
- [ ] SEO: 메타 태그, 구조화된 데이터 적용

### 정성적 지표
- [ ] 프로젝트의 핵심 가치가 명확히 전달되는가?
- [ ] 주요 기능이 이해하기 쉽게 설명되었는가?
- [ ] 시각적으로 매력적인가?
- [ ] 사용자가 다음 행동(깃헙 방문, 데모 체험 등)을 취하고 싶은가?

### 측정 방법
- Google Analytics (선택적)
- GitHub Stars/Forks 증가
- 이슈/PR 증가
- 사용자 피드백
